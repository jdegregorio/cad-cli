## Requirements specification for `cad-cli`

**Important**: Keep each requirement status up to date during development. Include comments that reference requirements in-code wherever possible.

### Functional requirements

| ID        | Requirement                                                                                                                                                           | Rationale                                                                                                       | Status    |
| --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- | --------- |
| CAD-F-001 | `cad-cli` shall provide a single stable command surface exposed as `cad`.                                                                                             | A unified deterministic CLI is a core architecture decision and keeps both human and agent usage simple.        | Completed |
| CAD-F-002 | `cad-cli` shall support the subcommands `build`, `render`, `compare`, `inspect`, and `package`.                                                                      | These are the agreed core deterministic operations for the project boundary of `cad-cli`.                       | Completed |
| CAD-F-003 | `cad build` shall accept a model source and optional parameters, execute the model, and emit standard output artifacts.                                              | Model generation must be deterministic and reusable outside Formloop.                                           | Completed |
| CAD-F-004 | `cad build` shall emit at minimum `STEP`, `GLB`, and metadata outputs for successful builds.                                                                         | The architecture explicitly standardizes STEP as authoritative and GLB as the presentation artifact.            | Completed |
| CAD-F-005 | `cad build` should optionally emit STL and a normalized source snapshot when requested.                                                                               | Optional export artifacts improve portability and inspection without changing the authoritative artifact model. | Completed |
| CAD-F-006 | `cad render` shall accept a GLB model plus a render specification and produce deterministic preview images.                                                          | Rendering is standardized through Blender with a stable render contract.                                        | Completed |
| CAD-F-007 | `cad render` shall output at minimum `front`, `back`, `left`, `right`, `top`, `bottom`, `iso`, and a composite render sheet.                                        | These views are the agreed default preview set for review and UI display.                                       | Completed |
| CAD-F-008 | `cad compare` shall compare two geometries and return structured overlap and delta outputs.                                                                           | Comparison is a first-class system capability used both in review and developer evals.                          | Completed |
| CAD-F-009 | `cad compare` shall support candidate-versus-ground-truth comparison, prior-revision comparison, and imported-reference comparison.                                  | Comparison is needed in more than one workflow, not only evals.                                                 | Completed |
| CAD-F-010 | `cad compare` shall return structured metrics including shared volume and exclusive volume for each input, where supported by the underlying geometry representation. | Deterministic overlap metrics are foundational for developer evals and high-confidence comparison.              | Completed |
| CAD-F-011 | `cad compare` shall support rigid alignment prior to comparison when inputs are not already in a common pose.                                                        | The agreed comparison approach separates alignment from boolean comparison.                                     | Completed |
| CAD-F-012 | `cad inspect` shall provide deterministic geometric inspection capabilities without requiring a second geometry input.                                                | Internal review loops depend on deterministic spot checks even when no ground truth exists.                     | Completed |
| CAD-F-013 | `cad inspect` shall support at minimum bounding box, overall dimensions, thickness, hole diameter, center distance, and volume queries.                              | These are the most useful inspection primitives for internal review and eval judges.                            | Completed |
| CAD-F-014 | `cad package` shall collect selected artifacts into a clean bundle for download or archival.                                                                         | Packaging is required for UI downloads, debugging, and eval artifacts.                                          | Completed |
| CAD-F-015 | `cad-cli` shall produce machine-readable output formats for commands that are intended to be consumed by Formloop or CI.                                             | Formloop and CI both need structured outputs, not only human-readable text.                                     | Completed |
| CAD-F-016 | `cad-cli` should provide a concise human-readable summary alongside machine-readable outputs for compare, inspect, and package operations.                           | Human operators and developers benefit from readable summaries during local development and debugging.          | Completed |
| CAD-F-017 | `cad-cli` shall preserve enough metadata to trace outputs back to the source model, parameters, and command invocation.                                              | Artifact traceability is a project-wide design goal.                                                            | Completed |
| CAD-F-018 | `cad-cli` shall support operation as a standalone tool outside Formloop.                                                                                             | Reusability by humans, CI, and future systems is an explicit boundary for the tool repo.                        | Completed |
| CAD-F-019 | All rendered preview images shall show the full geometry, fit to the size of the part, without clipping the model at image boundaries.                              | Preview renders are only useful if the full part is visible and framed consistently for review.                 | Completed |
| CAD-F-020 | Rendered previews shall include clear datum-oriented orthographic views plus at least one standard angled view such as isometric.                                    | Engineering review needs both datum-aligned inspection views and an angled view for overall shape comprehension. | Completed |
| CAD-F-021 | Rendered previews shall emphasize silhouette and feature edges enough for human verification of faces, holes, and other surface transitions.                        | Review renders need edge readability, not only soft shaded massing.                                             | Completed |
| CAD-F-022 | `cad build` shall accept an optional `--python PATH` flag that evaluates the model callable in the supplied interpreter while cad-cli owns presentation exports and metadata. | Operators need to evaluate models against their own project venv (matching their test environment) without forcing cad-cli's pinned dependencies. | Completed |

### Non-functional requirements

| ID         | Requirement                                                                                                                                          | Rationale                                                                                     | Status    |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- | --------- |
| CAD-NF-001 | `cad-cli` shall behave deterministically for the same inputs, environment, and configuration to the greatest practical extent.                      | Determinism is essential because `cad-cli` is the project’s repeatable tool substrate.        | Completed |
| CAD-NF-002 | `cad-cli` shall return explicit error codes and actionable error messages for command failures.                                                     | Reliable automation and debugging require machine-detectable failures and useful diagnostics. | Completed |
| CAD-NF-003 | `cad-cli` shall be operable in non-interactive environments such as CI runners and agent runtimes.                                                  | Both Formloop and GitHub Actions need headless execution.                                     | Completed |
| CAD-NF-004 | `cad-cli` shall avoid any dependency on a GUI for normal operation.                                                                                 | The agreed architecture relies on CLI-first automation and background rendering.              | Completed |
| CAD-NF-005 | `cad-cli` shall complete commands within practical time bounds for iterative design usage, with performance measured and tracked during development. | Iteration speed matters for both agent loops and local development.                           | Completed |
| CAD-NF-006 | `cad-cli` shall write outputs to explicit caller-provided or well-defined output locations.                                                         | Predictable output paths simplify orchestration, packaging, and debugging.                    | Completed |
| CAD-NF-007 | `cad-cli` shall support versioned schemas or documented formats for machine-readable outputs.                                                       | Stable integration with Formloop and CI requires schema discipline.                           | Completed |
| CAD-NF-008 | `cad-cli` shall log enough execution detail to support debugging without requiring access to internal source code.                                  | The tool is intended to be usable as a separate repo and shared dependency.                   | Completed |
| CAD-NF-009 | `cad-cli` should degrade gracefully when a requested operation is unsupported for a given input type, and shall report that limitation clearly.     | Some operations may vary by artifact type or geometry representation.                         | Completed |
| CAD-NF-010 | `cad-cli` shall be suitable for automated regression testing as part of CI/CD.                                                                      | The broader project explicitly expects batch eval and CI integration.                         | Completed |

### Design and technical constraint requirements

| ID        | Requirement                                                                                                                         | Rationale                                                                                         | Status    |
| --------- | ----------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- | --------- |
| CAD-D-001 | `cad-cli` shall use build123d as the primary CAD authoring backend for core model generation workflows.                             | build123d is the selected primary backend for the project.                                        | Completed |
| CAD-D-002 | `cad-cli` shall treat STEP as the authoritative CAD artifact.                                                                       | STEP is the agreed source-of-truth artifact for CAD and comparison.                               | Completed |
| CAD-D-003 | `cad-cli` shall treat GLB as the standard presentation artifact for rendering workflows.                                            | GLB is the agreed handoff format between modeling and rendering.                                  | Completed |
| CAD-D-004 | `cad-cli` shall use Blender as the standard rendering backend for `cad render`.                                                     | Blender is the agreed single renderer for consistent presentation.                                | Completed |
| CAD-D-005 | `cad-cli` shall keep rendering logically separate from geometry authoring and shall not treat Blender as a geometry source of truth. | The architecture explicitly separates CAD authority from rendering.                               | Completed |
| CAD-D-006 | `cad-cli` shall implement geometric comparison so that alignment and overlap computation are separable stages.                      | The agreed comparison approach treats alignment as distinct from exact boolean comparison.        | Completed |
| CAD-D-007 | `cad-cli` should prefer exact solid comparison where possible and use mesh-based fallbacks only when necessary.                     | Exact CAD-solid comparison provides better evaluation fidelity than mesh-only comparison.         | Completed |
| CAD-D-008 | `cad-cli` shall preserve clean module boundaries internally even though it is packaged as one repo and one CLI product.             | The user explicitly simplified repo structure, but functional boundaries still matter internally. | Completed |
| CAD-D-009 | `cad-cli` shall reserve `cad validate` as an optional future extension point rather than a core v1 command.                         | 3D printing validation is explicitly out of scope for the core version.                           | Completed |
